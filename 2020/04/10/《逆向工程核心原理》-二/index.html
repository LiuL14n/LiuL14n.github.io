PE文件格式

介绍

前期引入1: x86都是32位的，IA-64都是64位的。64位Windows需要做的只是修改PE格式的少数几个域。这种新的格式被称为PE32+。

前期引入2：讲解PE文件时经常出现映像（Image)这一术语。PE文件加载到内存时，文件不会原封不动地加载，而要根据节区头中定义的节区起始地址、节区大小等加载。因此，磁盘文件中的PE与内存中的PE具有不同形态。将装载到内存中的形态称为“映像”以示区别。

  1. 如何加载到内存、从何处开始运行、运行中需要的DLL有哪些、需要多大的栈和堆内存等，大量信息以结构体的形式存储在PE头中。
  2. 计算机中，为了提高处理文件、内存、网路包的效率，使用“最小基本单位”这一概念，PE文件中也类似。文件/内存中节区的起始位置应该在各文件/内存最小单位的倍数位置上，空白区域将使用NULL填充。
  3. VA是进程虚拟内存的绝对地址，RVA是指从某个基准位置(ImageBase)开始的相对地址，VA = RVA + ImageBase。 PE头内部信息大多以RVA的形式存在。

PE头

DOS头

微软为了兼容DOS文件，在PE头最前面加了一个IMAGE_DOS_HEADER结构体。

IMAGE_DOS_HEADER大小为64字节，其中有两个需要知道的重要成员。

word e_magic: DOS签名(signature, ASCII值 MZ)

long e_lfanew: 指示NT头的偏移（根据不同文件可以拥有不同的值）

以本书附件notepad.exe为例，用010 editor打开，可以看到010editor非常人性化的将DOS头标记出来了，当鼠标点击IMAGE_DOS_HEADER结构体的成员的时候，还能具体到成员在文件中的位置。

而且各个成员的值、起始位置、所占字节数以及注释等都可以显示出来，不过与微软给定的结构体成员的名字不同的是，在010editor里显示的成员名字更具有可读性。

DOS存根

DOS存根（stub）在DOS头下方，是个可选项，且大小固定不变（即使没有DOS存根，文件依然能够正常运行）。DOS存根由代码和数据混合而成。

比较有趣的是：文件偏移40~4D区域是16位的汇编，由于这个程序被识别成了PE文件，所以Windows在运行这个程序的时候不会执行这段汇编。通过DOS调试器运行该程序的话可以执行这段汇编代码。

NT头

即IMAGE_NT_HEADERS

它有三个成员

第一个成员是签名结构体：其值为50450000h(“PE”00)，其值固定不变。

第二个成员是文件头(File Header)

第三个成员是可选头(Optional header)

NT头的大小为F8。

NT头：文件头

即IMAGE_FILE_HEADER

文件头里有四个成员较为重要，若它们设置不正确

  1. NumberOfSections：由于PE文件把代码、数据、资源等一句属性分类到各个节区中储存，所以NumberOfSections用来指出文件中存在的节区数量。该值一定要大于0，其当定义的节区数量与实际节区不同时，将发生运行错误。

  2. SizeOfOptionalHeader：可选头大小。

尽管可选头是由C语言写成，其大小已经确定（32位文件是224,64位文件是240），但是windows的PE装载器需要查看SizeOfOptionalHeader的值来确定可选头的大小。

  1. Characteristics：该字段用于标识文件的属性，文件是否是可运行的状态、是否为DLL文件等信息，以bit OR形式组合起来。

以下是在winnt.h中的定义。

需要记住0002h 与 2000h这两个值，分别表示文件是可执行的 与 文件是个DLL

需要注意的是：PE文件该字段的值有可能不是0002h，例如obj文件以及resource DLL文件等。

而且较为诡异的是，明明是word类型，但是winnt.h里面却只提供了15个标志位，可以看到从0x0020直接蹦到了0x0080，略过了0x0040,在010 editor里面也是，也就是说第七位是缺失的，那么就可以把它默认为0了。

010 editor能对这个字段以及里面具体的位进行查看、修改等，非常方便。

NT头：可选头

可选头是PE头结构体中最大的，32位文件是224字节，64位文件是240字节。

如下是32位的可选头的定义。

以下结构体成员是需要关注的，这些成员的值是文件运行必需的，设置错误将导致文件无法正常运行。

  1. Magic：为IMAGE_OPTIONAL_HEADER32时，此值为10B,为IMAGE_OPTIONAL_HEADER64时，此值为20B。
  2. AddressOfEntryPoint：它持有EP的RVA值，能指出程序最先执行的代码的起始地址，相当重要。
  3. ImageBase：进程的虚拟内存的范围是0-FFFFFFFF(32位系统)，PE文件被加载到如此大的内存中时，ImageBase指出文件的优先装入地址。EXE和DLL被装载到用户内存的0-7FFFFFFF中，SYS文件被载入到内核内存的80000000-FFFFFFFF中。通常情况下，用VB/VC++/Delphi写好的EXE文件的ImageBase值为00400000，DLL文件的ImageBase值为10000000（当然也可以指定为其他值）。执行PE文件时，PE装载器先创建进程，再将文件载入内存，然后设置EIP = ImageBase + AddressOfEntryPoint
  4. SectionAlignment、FileAlignment：FileAlignment指定了节区在磁盘文件中的最小单位，SectionAlignment指定了节区在内存中的最小单位。
  5. SizeOfImage：加载PE文件到内存时，SizeOfImage指定了PE Image在虚拟内存中所占空间的大小。一般而言，文件的大小与加载到内存中的大小是不同的（节区头中定义了各节装载的位置与占有内存的大小）。
  6. SizeOfHeaders：该值指出了整个PE头的大小，该值必须是FileAlignment的整数倍。第一节区所在位置与SizeOfHeaders距文件开始偏移的量相同。可以以此值作为PE文件第一节区的文件偏移量。
  7. Subsystem：该值用来区分系统驱动文件（ *.sys） 与 普通的可执行文件（ *.exe、 *.dll）。
值含义备注
1Driver文件系统驱动文件
2GUI文件窗口应用程序
3CUI文件控制台应用程序
  1. NumberOfRvaAndSizes： NumberOfRvaAndSizes用来指定DataDirectory（IMAGE_OPTIONAL_HEADER32结构体最后一个成员）结构体数组的大小。虽然结构体定义中明确指出了结构体数组大小为16（#define IMAGE_NUMBEROF_DIRECTORY_ENTRIES 16）但是PE装载查看的NumberOfRvaAndSizes的值来识别数组的大小（即数组大小也可能不为16）。
  2. DataDirectory：是由IMAGE_DATA_DIRECTORY结构体组成的数组，数组的每项都有被定义的值，这里的Directory可以理解成结构体数组。

可以看到每个Directory都由RVA和SIZE组成。

其中EXPORT / IMPORT / RESOURCE / TLS 较为重要。

节区头

引入：PE文件格式的设计者们决定把具有相似属性的数据统一保存在一个被称为节区的地方，然后需要把各节区属性记录在节区头中（文件/内存的起始位置、大小、访问权限等）。节区头是由IMAGE_SECTION_HEADER结构体组成的数组，每个结构体对应一个节区。

有几个重要成员。

项目含义
VirtualSize内存中节区所占大小
VirtualAddress内存中节区起始地址（RVA）
SizeOfRawData磁盘文件中节区所占大小
PointerToRawData磁盘文件中节区起始位置
Characteristics节区属性（bit OR）

Q：节区头成员 VirtualAddress 是内存中节区头的起始地址（RVA），VirtualAddress 不就是 VA 吗？为什么要叫 RVA 呢？

A：“使用 RVA 值来表示节区头的成员 VirtualAddress”，这样理解就可以。节区头结构体（IMAGE_SECTION_HEADER）的 VirtualAddress 成员与虚拟内存地址（VA，VirtualAddress）用的术语相同才引起这一混乱。“节区头成员 VirtualAddress 指的是虚拟内存中相应节区的起始地址，它以 RVA 的形式保存在结构体中”，如此理解即可。

VirtualAddress与PointerToRawData不带有任何值，分别由（定义在IMAGE_OPTIONAL_HEADER32）SectionAlignment和FileAlignment确定。
VirtualSize与SizeOfRawData一般具有不同的值（即磁盘文件中节区的大小与加载到内存中的节区大小是不同的）。
Characteristics显示值的组合（bit OR）而成

Name成员不像C语言中的字符串一样以NULL结束（没有“必须使用ASCII值”的限制，对Name未明确规定），所以可以放入任何值，甚至可以填充NULL值。（数据节区的名称也可以叫做.code）

RVA to RAW

PE文件加载到内存时，每个节区都要准确完成内存地址与文件偏移的映射。这种映射便称为RVA to RAW，实现这个过程需要经过以下的步骤。
1.查找RVA所在节区。
2.使用公式计算出文件偏移。
根据IMAGE_SECTION_HEADER结构体，换算公式如下。
RAW - PoniterToRawData = RVA - VirtualAddress
RAW = RVA - VirtualAddress + PointerToRawData

注意：如果某个节区的VirtualSize的值比SizeOfRawData要大，则会出现“无法定义与RVA相对应的RAW值”的错误情况。

IAT

（Image Address Table）：本章的重头大戏！是掌握Windows操作系统的根基。简言之，IAT是一种表格，用来记录程序正在使用哪些库中的哪些函数。

DLL

加载DLL的方式有两种：一种是“显式链接”（Explicit Linking），程序使用DLL时加载，使用完毕后释放内存。另一种是“隐式链接”（Implicit Linking），程序开始时遍一同加载DLL，程序终止时，再释放占用的内存。IAT提供的机制便与隐式链接有关。

Q: 使用如CALL DWORD PTR DS:[1001104]的形式的原因（1001104中存有某DLL中某函数的地址）？

A：由于运行环境不同、DLL版本可能不同、DLL的重定位、PE头中表示地址时用的是RVA而非VA，所以在执行文件时，由PE装载器将该函数的地址写到1001104的位置。

IMAGE_IMPORT_DESCRIPTOR

该结构体记录着PE文件要导入哪些库文件。

需要注意这三个RVA。

执行一个普通程序时往往需要导入多个库，导入多少库就存在多少个IMAGE_IMPORT_DESCRIPTOR结构体，这些结构体形成了数组，且结构体数组最后以NULL结构体结束。

INT与IAT是长整型（4个字节数据类型）数组，以NULL结束（未另外明确指出大小）。

INT中各元素的值为IMAGE_IMPORT_BY_NAME结构体指针（有时IAT也有相同的值）。

INT与IAT的大小应相同。

PE装载器把导入函数输入至IAT的顺序：

  1. 读取IID的Name成员，获取库名称字符串（eg：kernel32.dll）

  2. 装载相应库：

    LoadLibrary(“kernel32.dll”)

  3. 读取IID的OriginalFirstThunk成员，获取INT地址

  4. 逐一读取INT中数组的值，获取相应IMAGE_IMPORT_BY_NAME地址（RVA）

  5. 使用IMAGE_IMPORT_BY_NAME的Hint（ordinal）或Name项，获取相应函数的起始地址：

    GetProcAddress(“GetCurrentThreadld”)

  6. 读取IID的FirstThunk（IAT）成员，获得IAT地址

  7. 将上面获得的函数地址输入相应IAT数组值

  8. 重复以上步骤4～7，知道INT结束（遇到NULL）

IMAGE_IMPORT_OPTIONAL_HEADER32.DataDirectory[1].VituralAddress的值即为IMAGE_IMPORT_DESCRIPTOR结构体数组的起始地址(RVA)，因此IMAGE_IMPORT_DESCRIPTOR结构体也被称为IMPORT Directory Table

RVA of IMPORT Directory 为 0x7604,通过RVA to RAW 可以得到 RAW = RVA(7604) - VirtualAddress(1000) + PointToRawData(400) = 6A04

可以看到对应的6A04的位置正好是IMAGE_IMPORT_DESCRIPTOR结构体数组的起始位置。

虽然size of IMPORT Directory为0xc8,但观察可以发现0x6a04 + 0xc8 却不等于user32.dll的末尾0x6AB7,多出来的那些即为NULL结构体。

来看comdlg32.dll的对应部分

其INT的RVA(OriginalFirstThunk)为0x7990对应RAW为0x6D90,是存放着各个函数名字的地址(RVA)的数组(即IMAGE_IMPORT_BY_NAME结构体指针数组)

以起始位置存放的0x7A7A为例，对应RAW为0x6E7A,可以看到函数名前面有WORD类型的Hint(000F)，即Ordinal，是库中函数固有的编号，函数名以\0结尾。

其IAT的RVA(FirstThunk)为0x12C4对应的RAW为0x6C4

可以看到INT和IAT大小相等。

这里IAT中的硬编码的值并没有实际意义，当该exe文件加载到内存的时候，准确的地址值会取代这些值。

来用OD载入看一下，可以发现新的地址50F44E77取代了硬编码的76324906（Win 10 系统），但是也有些情况硬编码的值和载入内存以后的值相同。

EAT

与IAT相对的则是EAT，与IAT能够拥有IMAGE_IMPORT_DIRECTOR数组不同的是，PE文件只有一个用来说明EAT的IMAGE_EXPORT_DIRECTORY结构体（因为PE文件可以同时导入多个库，但只能导出一个）

项目含义
NumberOfFunctions实际导出函数的个数
NumberOfNames导出函数中具名函数的个数
AddressOfFunctions导出函数地址数组,个数=N O F
AddressOfNames函数名称地址数组,个数=N O N
AddressOfNameOrdinalsOrdinal地址数组,个数= N O N

并非所有的导出函数都有名字，有时候某些函数不会对外公开函数名，仅公开函数的固有编号，所以才会导致上面表格中数组个数取决于不同的值的情况。
